# V0.14 Canal Runtime Test Build

V0.14 wires the Canal adapter into executable Edge paths.

## What Changed

- Edge `sync-agent run` starts `edge-cdc-canal` when `cdc.type: canal`.
- `edge-cdc-canal` fetches a Canal batch and publishes normalized `SyncEvent` messages to local RabbitMQ.
- Canal ACK happens only after all publish operations succeed, or after a batch is safely suppressed.
- `canal-publish-once` provides a one-shot test command.

## One-Shot Test

```powershell
go run ./cmd/sync-agent canal-publish-once `
  -config configs/edge.example.yaml `
  -rules configs/sync-rules.example.yaml `
  -amqp-url amqp://sync:password@127.0.0.1:5672/edge-sync
```

Expected output:

```text
canal batch published action=published event_id=<id> count=<n>
```

If Canal has no row events:

```text
canal batch empty
```

## Long-Running Edge Runtime

```powershell
go run ./cmd/sync-agent run `
  -config configs/edge.example.yaml `
  -rules configs/sync-rules.example.yaml
```

When `cdc.type` is `canal`, Edge mode runs:

- `edge-cdc-canal`
- `edge-upload`
- `edge-downlink`

## Reliability Rule

The Canal batch flow is:

```text
Fetch Canal batch
  -> Convert to ChangeEvent
  -> Loop suppress
  -> Normalize to SyncEvent
  -> Publish to local RabbitMQ, or safely suppress
  -> Save offset
  -> ACK Canal batch
```

ACK before publish is forbidden. Suppressed replay/ignored rows are considered safely processed and must also advance the offset.

## Current Limits

- Lab E2E still uses stub CDC unless Canal Server is added to the lab compose.
- Column values from Canal are currently raw strings or `nil`.
- This version does not yet install or configure Canal Server automatically.
