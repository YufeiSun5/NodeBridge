# V0.30 Retry Closure

## Summary

V0.30 收口失败重试最小闭环：失败 ACK 可查询、单条或批量标记重试、pending 事件可重新下发，死信队列可只读预览。业务数据仍通过 RabbitMQ 下发队列进入 Edge Apply，成功后继续依赖 `sync_apply_log` 幂等。

## New Backend API

Wails:

- `RetryFailedEvent(req)`：单条失败 ACK 标记为 `PENDING`。
- `RetryFailedEvents(req)`：批量失败 ACK 标记为 `PENDING`。
- `GetDeadLetters(req)`：预览死信队列消息，读取后立即 requeue。

CLI:

```powershell
SyncAgent.exe failed-events -config configs/lab/server.local.yaml -limit 10
SyncAgent.exe retry-event -config configs/lab/server.local.yaml -event-id <event_id> -target-node-id <node_id>
SyncAgent.exe retry-failed-batch -config configs/lab/server.local.yaml -limit 100
SyncAgent.exe replay-pending-once -config configs/lab/server.local.yaml
SyncAgent.exe dead-letters -config configs/lab/server.local.yaml -queue server.dead.q -limit 20
```

## E2E

```powershell
powershell -NoProfile -ExecutionPolicy Bypass -File .\scripts\lab-retry-e2e.ps1 -SkipPrepare
```

验证内容：

- Server `sync_event_log` 存在可重放 payload。
- Server `sync_ack_log` 存在 `FAILED` 目标节点记录。
- `retry-failed-batch` 将失败 ACK 标记为 `PENDING`。
- `replay-pending-once` 重新发布到目标 Edge downlink queue。
- Edge B `consume-downlink-once` 成功 Apply。
- `sync_ack_log` 最终变为 `SUCCESS`。
- `dead-letters` 预览 `server.dead.q` 后消息仍保留。

## Limits

- 本版是最小闭环，不做复杂人工修复界面。
- 死信预览会读取消息再 requeue，适合诊断，不适合高频轮询。
- 批量重试只按 `sync_ack_log.status=FAILED` 选取前 N 条，不做复杂筛选。
- 前端还需要接入批量重试和死信预览按钮。
