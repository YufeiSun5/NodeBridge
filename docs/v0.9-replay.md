# V0.9 Event Replay

V0.9 adds the first durable replay path for Server downlink dispatch.

## Scope

- Server ingress persists the raw `SyncEvent` payload in `sync_event_log.event_payload`.
- `retry-event` marks a failed `(event_id, target_node_id)` as `PENDING`.
- `replay-pending-once` reads one pending replay and republishes it to `server.dispatch.x`.
- Successful replay updates `sync_dispatch_log` and `sync_ack_log` to `SUCCESS`.

## Commands

```powershell
sync-agent retry-event `
  -config configs/server.example.yaml `
  -event-id evt-001 `
  -target-node-id edge-002

sync-agent replay-pending-once `
  -config configs/server.example.yaml `
  -amqp-url $env:NODEBRIDGE_RABBITMQ_URL
```

If `-amqp-url` is omitted, the command uses `rabbitmq.server_url` from the config.

## Current Limits

- Replay is single-step for development verification.
- Automatic background replay scheduling is not enabled yet.
- Existing databases need the `event_payload` column on `sync_event_log`.
- RabbitMQ remains configurable; Docker RabbitMQ is only for development tests.
